WHY · 这套架构要解决什么问题

业务方提"为什么 GMV 跌了 20%" → 不能只给一个数字，要拆到渠道/品类/人群/时间维度给出可落地结论。 传统做法要分析师写 SQL+出报告（天级），现在用 LLM Agent + Skill 把归因方法论固化，5-10 分钟自助完成，过程全可追溯。

前端 · auth/web
Frontend · React 19 SPA
用户入口

浏览器端 SPA，由 insight-agent 后端静态托管。负责登录态、对话列表、WebSocket 流式渲染、报告查看。

  • OAuth2 + PKCE 登录，跳转 auth 服务换取 access_token
  • WebSocket 流式接收 assistant 文本 + 工具调用 + 工具结果
  • 附件下载 / 历史消息分页 / 草稿对话管理
  • 权限管理页：用户/角色/权限三列联动筛选 + 关系编辑

insight-agent · 核心智能体
Core Agent · FastAPI + deepagents
项目主体

基于 deepagents (LangGraph) 框架，装配 LLM + 工具 + Skill + 工作区，提供对话、流式聊天、文件落盘、热更新 Agent 实例。

  • 3 层记忆：MySQL 消息流 + 上下文压缩 + 对话级工作区文件
  • WebSocket 长连接，单轮失败不影响连接，协作式取消
  • 5 个 Skill：insight(归因) / docx / pdf / pptx / xlsx
  • MCP 工具：tavily_search 等外部能力即插即用
  • 自动重试 + 上下文压缩（seq_offset 换算对齐 DB 序号）
  • 热更新：POST /api/reload 不重启重建 Agent

data-agent · 数据查询
Text-to-SQL Service
数据接入

被 insight-agent 的 db_query 工具以 SSE 方式调用的"取数大脑"，把自然语言问句翻译成 SQL 并执行。

  • Elasticsearch 索引数仓元数据，不把 schema 写进 prompt
  • LLM 生成 SQL → 查 MySQL → 表格落 CSV / 非表格落 JSON
  • 返回 pandas_read_hint 方便 Agent 后续读取
  • SSE 流式输出，与 insight-agent 异步架构对齐
FastAPIElasticsearchLangChainSSE
🔐
auth · 统一认证
OAuth2 + PKCE Identity
身份中心

项目所有对外 HTTP 服务的统一身份中心。前后端分离 + 多端场景的标准选择。

  • OAuth2 Authorization Code + PKCE(S256)，防授权码被截
  • introspection 端点实时校验 token，支持立即撤销
  • 用户 / 角色 / 权限 / 关系全套 CRUD，管理员 RBAC
  • 密码 / 邮箱变更级联使旧 session/token 失效

基础设施
Infra · MySQL · Redis · ES · LLM
依赖

项目依赖的外部 / 内部基础设施。docker-compose 一键拉起 MySQL + Redis。

  • MySQL：存 conversation / message / context_compaction

  • LLM 服务：兼容 OpenAI 协议，可本地或云端部署
