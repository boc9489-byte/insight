🛒 商家工作台 × AI 调度 · 整体架构

面向 B 端商家（淘宝商家版风格）· 归因分析为主，问数系统为底层工具
架构版本 v1.0 · 2026.06 · ← 主文档 · → 总架构图

🎯 核心思想：归因为主，问数为辅

商家在日常工作台里提的问题 90% 是"为什么"型问题（"GMV 跌了为什么？"、"退款率涨了为什么？"）， 这些问题 都先进入归因分析（insight-agent） 处理；归因过程里会反向调用 db_query 工具 → 该工具内部走 SSE 到 问数系统（data-agent） 取数。

只有 "是什么/多少"型问题（"上个月 GMV 多少？"、"今天的访客数多少？"）才直接走问数，秒级返回单值。

🤖
意图识别 + 优先级路由
AI Assistant for Merchants
调度
  • 第 1 优先级：归因分析 — 任何涉及"为什么/怎么拆解/怎么改善"的问题都进 insight-agent
  • 第 2 优先级：直接问数 — 仅"是多少/查个数"类问题直达 data-agent（db_query 工具背后就是它）
  • 第 3 优先级：智库 / 操作 / 工单 — 知识问答 / 商家工作台原生 API / 危机处理工单
  • 统一鉴权复用本项目 auth 服务，商家一次登录全平台通行
▼ 主链路：先归因
★
归因分析（核心引擎 · 最高优先级）
insight-agent · 商家问题的"主入口"
🧠
归因分析 Agent
insight-agent · 多维拆解 + 业务结论 + HTML 报告
★ 核心引擎
商家 90% 的问题都先到这里。Agent 通过 deepagents + insight Skill 把"指标异常"拆解为可落地的"渠道/品类/人群/时间"四维归因报告。
  • 触发场景：GMV 跌 / 退款率涨 / 投放效果差 / 选品 / 备货 / 差评危机...
  • 内部动作：insight Skill 工作流 → 补基线 → 多维拆解 → 异常识别 → HTML 报告
  • 依赖：执行过程中会反向调 db_query 工具取数（见下方）
  • 输出：HTML 报告 + 一键操作建议 + 工单触发
  • 响应：分钟级（5-10 分钟）
▼ 归因过程会反向调 db_query 取数

结果聚合
  • HTML 归因报告（多维拆解 + 业务建议）—— 90% 来自归因分析
