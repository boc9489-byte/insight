# 数据生成
## 使用方式
修改 [./core/init_db.py](./core/init_db.py#L195) 下方数据库连接配置  

```bash
uv sync                         # 安装依赖
uv run core/init_db.py          # 初始化数据库（创建表结构）
uv run -m core.generators.main  # 生成数据
```

## 调整数据规模
关键目标量集中写在 [./core/generators/settings.py](./core/generators/settings.py)：

```python
USER_INITIAL_COUNT = 1_000
USER_FINAL_COUNT = 3_000
SPU_TARGET_COUNT = 500
ORDER_DETAIL_TARGET_COUNT = 100_000
PAGE_VIEW_EVENTS_PER_USER = 10
```

需要进一步缩小或放大时，直接修改 `settings.py` 中对应配置后重新生成数据。

| 配置项                      | 默认值  | 说明                   |

| `USER_INITIAL_COUNT`        | 1,000   | 初始用户数             |

| `SPU_TARGET_COUNT`          | 500     | SPU 目标数             |

## 业务规则
### 1. 时间分布
- **时间范围**: 近三年数据（三年前 至 今天）
- **促销高峰**: 
  - 618期间（6月1日-6月18日）订单量 × 8
  - 双11期间（11月1日-11月11日）订单量 × 10
  - 双12期间（12月1日-12月12日）订单量 × 5
- **日常分布**: 工作日 vs 周末，白天 vs 夜间有合理差异

### 2. 订单状态流转
```
创建订单
  ├── 15% → 未支付（终态）
  └── 85% → 已支付
        ├── 5% → 支付后取消（终态）
        └── 95% → 已发货
              ├── 2% → 拒收 → 退款
              └── 98% → 已签收
                    └── 10% → 申请退款（约10%退款率）
```

### 3. 优惠分摊规则
- **活动参与率**: 40%订单参与促销活动
- **优惠券使用率**: 40%订单使用优惠券
- **分摊逻辑**: 按订单金额比例分摊到各个明细行
- **重叠逻辑**: 活动和优惠券可重叠

### 4. 运费与税费规则
- **订单级运费**: 运费先按订单级生成，再按订单金额比例分摊到明细行
- **包邮门槛**: 不同一级类目采用不同包邮门槛，低客单价类目更容易出现非零运费
- **店铺修正**: 自营店更容易包邮，跨境店铺通常收取更高运费
- **税费规则**: 跨境订单按订单金额的一定比例生成税费并分摊到明细

### 5. 库存扣减逻辑
```
下单 → 锁定库存
  ├── 支付成功 → 扣减库存，释放锁定
  ├── 支付超时/取消 → 释放锁定，恢复可用
  └── 退款成功 → 恢复库存（退货场景）
```

### 6. 数据关联规则
- 用户口径：用户维度采用标准拉链存储，包含 `start_date`、`end_date`、`is_current` 字段；首版结束日期默认为 `9999-12-31`，用户属性变更时关闭旧版本并新增当前版本
- 低频维表口径：店铺、类目、品牌、支付方式、物流公司、地理区域同样采用标准拉链存储；首版从生成开始日期生效，主数据变更时关闭旧版本并新增当前版本
- 商品口径：SPU 和 SKU 维度采用标准拉链存储，首版从商品首次可见日期生效，后续仅在商品属性变更时关闭旧版本并新增当前版本
- 营销口径：促销活动和优惠券维度采用每日快照存储，在活动有效期或优惠券可用周期内按天生成快照
- 批次1静态维度：只读取 `core/generators/seeds/*.json` 冻结种子，不在运行时随机生成
- 每个订单必须有合法的用户、店铺、SKU
- 评论必须关联已签收的订单，且评论时间在签收时间之后
- 退款必须关联已支付的订单
- 支付必须关联已创建的订单，且支付事实按订单粒度生成，不关联订单明细ID
- 发货必须在支付之后

### 7. 脱敏规则
- **手机号**: 138****8888 格式
- **邮箱**: k***@gmail.com 格式
- **地址**: 保留省市，详细地址模糊化
- **姓名**: 张**，李** 格式

## 实现要点

### 技术选型
- **数据库操作**: SQLAlchemy
- **静态主数据**: 本地 JSON 冻结种子
- **批量插入**: 每批次 10000 条，减少数据库往返

### 当前实现说明
- **批次1**: 用户维表和 6 张低频维表都维护拉链版本
- **历史口径**: 涉及历史日期的事实或维度生成，应按业务日期关联拉链表有效版本（`start_date <= 业务日期 <= end_date`）；只有明确需要当前口径时才读取 `is_current=1`
- **批次4支付表**: `dwd_fact_trade_pay_detail_di` 按订单粒度生成，仅保留 `order_id` 关联
- **批次4运费**: 运费按订单级生成后分摊，包邮逻辑按类目、店铺类型、跨境属性和订单金额综合决定
- **主调度**: 批次1-3串行执行；批次4和批次5在依赖满足后并行执行
- **批次5跳过规则**: 仅当加购、收藏、页面访问、搜索四张行为事实表都已有数据时才跳过；若只生成了部分表，会直接报错提示先清理

### 批次1来源
- **店铺/类目/品牌**: 基于真实电商平台公开信息整理的冻结种子
- **支付方式/物流公司**: 基于主流电商平台常见能力与行业公开名单整理
- **地理区域**: 基于中国省/市/区真实行政区划名称整理
- **种子维护**: 直接编辑 `core/generators/seeds/*.json`
